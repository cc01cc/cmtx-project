用统一的判断框架处理 API 设计争议，避免每次都从头讨论命名、签名和兼容性。

## When to Use This Skill

在以下场景使用本 skill：

- 用户想重命名函数、类型、接口或导出项
- 用户想把位置参数迁移为 options object
- 用户想拆分一个过于模糊的 API，或合并过度碎片化的 API
- 用户质疑命名是否表达真实语义
- 用户需要评估兼容性、迁移成本和调用方影响

不要把本 skill 用于纯内部私有函数的小改名，除非该命名会影响公共设计判断。

## Required Inputs

尽量先明确：

1. 当前 API 定义和典型调用方式。
2. 主要问题是命名不清、签名不稳、职责过多还是调用易错。
3. 是否为 public API。
4. 是否允许 breaking changes。
5. 现有调用方分布和迁移约束。

## Analysis Workflow

### 1. 先定位真正的问题类型

区分下面几类问题：

- 命名问题：名字不表达语义或模式不统一
- 签名问题：参数顺序难记、可选参数太多、布尔参数语义弱
- 职责问题：一个函数做了两种以上稳定变化原因的事情
- 兼容问题：当前设计已被广泛使用，不能轻易改动

不要把所有问题都粗暴归结为“改成 options object”。

### 2. 检查命名一致性

优先检查：

- 同类函数是否遵循同一动词模式
- 布尔判断函数是否命名为 `is*` / `has*` / `can*`
- 名称是否准确表达输入约束和输出语义
- 近义函数是否只是名称不同但逻辑重复

如果只是去重命名，优先减少重复概念，而不是继续增加别名。

### 3. 检查签名设计

以下情况通常说明签名需要重构：

- 连续多个同类型位置参数，易传错顺序
- 存在多个布尔参数
- 参数之间强耦合，但从签名上看不出来
- 调用方经常只传最后几个可选参数

以下情况更适合继续保持位置参数：

- 参数数量少且语义强
- 高频调用，且当前签名非常直观
- 公开 API 已广泛采用，改动收益有限

### 4. 判断是否拆分函数

满足以下任意两条，通常适合拆分：

- 同一函数根据 `format`、`mode`、`type` 执行两套明显不同的行为
- 文档示例需要大量条件说明
- 返回值或副作用在不同模式下差异很大
- 调用方经常只需要其中一种行为

如果拆分后只会制造更多碎片 API，则不要拆。

### 5. 评估兼容策略

按风险从低到高考虑：

1. 保留旧 API，内部转调新实现。
2. 保留旧签名并标记废弃。
3. 提供过渡期 alias 或 overload。
4. 直接替换所有调用方并执行 breaking change。

选择策略时，必须说明：

- 为什么值得改
- 谁会受影响
- 如何迁移
- 何时删除兼容层

## Decision Heuristics

- 如果调用错误主要来自参数顺序，优先考虑 options object。
- 如果主要问题是职责混合，优先拆函数，而不是只改参数名。
- 如果只是命名不清，不要借机做超范围重构。
- 公共 API 的每一次改动都要以调用方成本衡量，而不是只看定义端是否更优雅。

常见失败模式与规避建议在参考文档中统一维护。

## Example Prompts

- 帮我评估这个 API 是否应该从位置参数改成 options object。
- 这个函数名不太对，帮我做命名和签名一致性分析。
- 我想把一个多模式函数拆成两个 API，给我方案和兼容策略。
