# TailwindCSS 官方文档优先策略

本参考文档用于约束 agent 在处理 TailwindCSS 任务时的信源选择、输出结构与排障优先级。

## 1. 信源优先级

按以下顺序建立事实：

1. Tailwind 官方文档：<https://tailwindcss.com/docs>
2. 官方安装与框架集成文档（如 Vite 集成）
3. 官方核心概念文档（类名扫描、函数与指令、主题、插件、升级）
4. 官方仓库与官方发布说明
5. 社区文章、博客、视频、模板仓库，仅作补充

## 2. 必查信息

处理 Tailwind 请求时，优先核对：

- 目标版本与文档版本是否一致
- 项目构建工具与官方推荐集成方式是否一致
- 类名检测机制与扫描源配置是否覆盖真实模板
- 是否使用了动态拼接类名导致不可检测
- 是否存在旧版本配置、旧插件或已弃用能力

## 3. 输出格式要求

输出建议遵循：

1. 先结论：给最小可行方案。
2. 再步骤：按官方文档顺序执行。
3. 再代码：提供可复制的最小示例。
4. 最后风险：说明版本差异和常见误区。

若无法确认官方依据，必须显式说明置信度与待确认项。

## 4. 常见误区

- 直接复用旧版 Tailwind 教程，忽略当前版本差异
- 依赖动态拼接 class 名称，导致样式生成不完整
- monorepo 场景下未显式处理扫描基路径
- 过度 `@apply`，导致样式组织与可维护性下降
- 将第三方 UI 模板的约定误认为 Tailwind 官方规范

## 5. 高价值实战经验

- 先跑通最小样式链路，再做主题和组件抽象
- 用“静态类名映射”替代字符串拼接，稳定构建产物
- 在多人协作中沉淀可复用模式：布局、状态、响应式、主题令牌
- 每次升级先做最小 PoC，再大规模迁移
- 排障时优先验证扫描范围、入口样式文件和构建日志

## 6. 何时必须再次检索官方文档

- 用户明确要求“最新版”“最新推荐方式”
- 涉及安装、迁移、兼容、插件、指令等易变化内容
- 构建行为与预期不一致，且怀疑版本差异
- 用户提供资料明显基于旧版文档或过期模板
